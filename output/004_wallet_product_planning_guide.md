# 월렛 서비스기획 실무 가이드

> **한 줄 소개**
> 월렛 서비스는 지갑 생성 버튼만으로 완성되지 않습니다. 키 관리 주체, 복구 방식, 지원 체인, 서명 권한, 수수료 부담, 원장 상태, CS 조회 기준을 함께 정의해야 운영할 수 있습니다.

`대상: 서비스기획자·PM·운영 담당자` · `난이도: 중급` · `범위: 월렛 기능, 스마트 월렛 UX, 재화 원장, 응모/교환, 온체인·오프체인, CS`

---

## 0. 기획자가 먼저 잡아야 할 관점

월렛 기능을 기획할 때 화면 흐름부터 그리면 중요한 정책이 빠지기 쉽습니다. 먼저 아래 질문에 답해야 합니다.

| 질문 | 왜 중요한가 |
|---|---|
| 키를 누가 관리하는가? | 분실, 탈퇴, 기기 변경, 사고 대응 범위가 달라집니다. |
| 어떤 체인을 지원하는가? | 주소 검증, 수수료, 전송 실패, CS 정책이 달라집니다. |
| 수수료는 누가 부담하는가? | 사용자 비용, 무료 이벤트, 남용 방지 정책이 필요합니다. |
| 어떤 재화를 내부 원장에 두고 무엇을 온체인화할 것인가? | 개인정보, 운영비, 외부 검증 가능성의 균형을 정해야 합니다. |
| 실패했을 때 어디까지 복구할 수 있는가? | 사용자 안내, 약관, 운영 도구, 개발 범위가 결정됩니다. |

> **핵심 원칙**
> “어떤 화면을 만든다”보다 “어떤 상태를 만들고, 변경하고, 취소하고, 추적할 수 있는가”가 더 정확한 요구사항입니다.

## 1. 출시 전 최소 산출물

월렛 또는 월렛이 결합된 리워드 서비스를 만들기 전에는 최소한 아래 문서가 필요합니다.

| 산출물 | 포함해야 할 내용 |
|---|---|
| 월렛 정책 정의서 | 키 관리 주체, 복구 방식, 지원 체인, 주소 생성/연결 정책 |
| 거래/서명 정책서 | 전송, 수신, dApp 연결, 토큰 승인, 승인 취소, 서명 문구 |
| 재화 정책서 | 포인트, 응모권, 쿠폰, 토큰, NFT의 지급·차감·만료·회수 기준 |
| 원장 설계서 | 내부 원장 ID, 온체인 해시, 외부 주문번호, 상태값, 보정 기준 |
| 운영/CS 매뉴얼 | 지연, 오입금, 미지급, 쿠폰 실패, 계정 변경, 부정사용 대응 |
| 법무/개인정보 검토표 | 선불성 재화, 경품, 모바일상품권, 개인정보, 위치정보, 약관 |

이 산출물이 없으면 화면은 만들어져도 운영 중 문제가 생겼을 때 “누가, 무엇을, 어디까지 처리할 수 있는지”가 불분명해집니다.

## 2. 월렛 코어 기능 정의

**핵심 메시지:** 월렛 기능 정의서는 화면 목록이 아니라 **권한, 상태, 실패 처리, 운영 조회 기준**을 포함해야 합니다.

| 기능 | 기획자가 정의할 것 | 빠지면 생기는 문제 |
|---|---|---|
| 지갑 생성/연결 | 신규 생성인지 기존 지갑 연결인지, 지갑 주소를 언제 사용자 계정에 묶는지 | 같은 사용자의 복수 주소, 주소 변경 CS 혼선 |
| 인증/복구 | 시드 문구, 소셜 로그인, 패스키, 고객센터 복구 중 어떤 모델인지 | 분실·탈퇴·기기변경 시 책임 소재 불명확 |
| 잔액/자산 조회 | 온체인 잔액, 앱 포인트, 쿠폰, 응모권을 어디서 조회하는지 | 화면 잔액과 실제 원장 불일치 |
| 전송/수신 | 지원 체인, 주소 검증, 수수료, 최소 전송액, 취소 가능 여부 | 오입금·오전송·수수료 민원 |
| dApp 연결/서명 | 연결 대상, 서명 문구, 승인 권한, 승인 취소 경로 | 사용자가 무엇을 허용했는지 모름 |
| 거래 내역/영수증 | 온체인 해시, 내부 원장 ID, 외부 API 주문번호를 함께 저장 | CS가 거래를 추적하지 못함 |

### 요구사항 작성 예시

```text
사용자는 이더리움 네트워크의 지갑 주소를 생성할 수 있다.
시스템은 생성된 주소를 사용자 계정에 1개 이상 연결할 수 있다.
운영자는 사용자 ID로 지갑 주소, 생성 시각, 연결 상태, 마지막 거래 해시를 조회할 수 있다.
사용자는 주소를 삭제할 수 없고, 비활성화만 요청할 수 있다.
```

**챕터 요약:** 월렛 코어 기능 정의서는 화면, 원장 상태, 권한 주체, 실패 처리까지 같이 써야 실무에서 작동합니다.

## 3. 스마트 월렛과 최신 UX

**핵심 메시지:** 월렛 UX는 “시드 문구를 외워라”에서 **패스키, 복구, 가스 대납, 묶음 실행**으로 이동하고 있습니다.

| 개념 | 비전공자 설명 | 서비스기획 영향 |
|---|---|---|
| 계정 추상화 | 지갑 계정 자체에 복구, 한도, 자동화, 수수료 대납 규칙을 넣는 방향 | 월렛을 단순 키 보관함이 아니라 정책이 있는 계정으로 설계 |
| ERC-4337 | 스마트 계정이 UserOperation을 보내고 번들러가 묶어 처리하는 구조 | 가스 대납, 배치 실행, 소셜 복구, 세션키 UX 검토 |
| EIP-7702 | 기존 EOA도 특정 코드에 권한을 위임해 스마트 월렛 기능을 쓸 수 있게 하는 업데이트 | 기존 지갑 사용자를 새 계정으로 강제 이전하지 않는 UX 가능 |
| 패스키 | 얼굴, 지문, PIN으로 로그인하는 공개키 기반 인증 | 로그인은 쉬워지지만 온체인 자산 복구와 권한 모델은 별도 설계 |
| Paymaster | 사용자 대신 가스비를 내주는 주체 | 무료 거래 이벤트, 첫 출금 지원, 남용 방지 한도 정책 필요 |
| 세션키 | 짧은 기간·작은 권한만 가진 임시 열쇠 | 반복 미션에서 매번 서명하지 않게 할 수 있으나 권한 만료 필수 |

### 기획자가 정해야 하는 것

| 항목 | 결정 질문 |
|---|---|
| 복구 | 사용자가 기기를 잃으면 누가 어떤 조건으로 복구할 수 있는가 |
| 한도 | 하루 전송 한도, 첫 출금 한도, 고위험 행동 제한이 있는가 |
| 가스 대납 | 어떤 거래까지 무료로 처리하고, 남용은 어떻게 막는가 |
| 세션 권한 | 어떤 기능을 몇 분/몇 회까지 자동 승인할 수 있는가 |
| 사용자 고지 | 사용자가 실제로 어떤 권한을 준 것인지 이해할 수 있는가 |

> **최신성 메모**
> 원본 문서는 Ethereum.org의 계정 추상화, ERC-4337, EIP-7702 설명과 FIDO/WebAuthn 자료를 기준으로 정리했습니다. 표준과 지갑 지원 범위는 변할 수 있으므로 출시 시점에는 공식 문서 재확인이 필요합니다.

**챕터 요약:** 최신 월렛 UX의 목표는 시드 문구를 숨기는 것이 아니라, 권한·복구·비용을 사용자가 이해 가능한 정책으로 바꾸는 것입니다.

## 4. 재화 정책과 원장 설계

**핵심 메시지:** 포인트, 응모권, 토큰, 상품권은 모두 재화입니다. 실무에서는 **생성 근거, 차감 순서, 유효기간, 회수 가능성**을 먼저 정합니다.

| 정책 항목 | 정해야 할 질문 | 실무 리스크 |
|---|---|---|
| 재화 종류 | 포인트, 캐시, 응모권, 쿠폰, NFT, 토큰을 구분했는가 | 회계·정산·약관·CS 기준 혼선 |
| 지급 근거 | 광고 시청, 설문, 구매, 이벤트 중 무엇으로 발생했는가 | 부정 참여·중복 지급 판단 어려움 |
| 차감 순서 | 유상/무상, 먼저 만료되는 재화, 이벤트 재화 중 무엇을 먼저 쓰는가 | 환불·민원·정산 오류 |
| 유효기간 | 만료 기준, 사전 고지, 연장 가능 여부는 무엇인가 | 소비자 분쟁 |
| 회수/취소 | 광고 취소, 구매 반품, 부정사용 시 회수 가능한가 | 보상비 누수 |
| 온체인화 기준 | 어떤 재화만 토큰/NFT로 남기고 무엇은 내부 DB에 둘 것인가 | 개인정보 노출·수수료·복구 어려움 |

### 원장 상태 예시

```text
issued      지급됨
available   사용 가능
reserved    교환 또는 응모 요청으로 임시 홀드
used        사용 완료
expired     유효기간 만료
reversed    취소 또는 회수
adjusted    운영자 보정
```

### 차감 정책 예시

```text
1. 만료일이 가장 빠른 무상 포인트
2. 만료일이 가장 빠른 유상 포인트
3. 이벤트 전용 포인트
4. 운영자 지급 보정 포인트
```

실제 차감 순서는 서비스 약관, 회계 기준, 환불 정책과 함께 검토해야 합니다.

> **전문가 검토**
> 현금성 포인트, 여러 가맹점에서 쓰는 선불성 재화, 모바일상품권은 전자금융거래법, 약관, 소비자보호 이슈가 생길 수 있습니다. 출시 전 법무·회계·개인정보 검토가 필요합니다.

**챕터 요약:** 재화 정책은 마케팅 문구가 아니라 정산, CS, 법무, 부정사용 탐지의 공통 언어입니다.

## 5. 경품 응모와 기프티콘 교환

**핵심 메시지:** 경품·기프티콘은 보상 화면이 아니라 **당첨/교환 상태 머신과 외부 API 운영**입니다.

| 항목 | 필요한 상태 | 체크 포인트 |
|---|---|---|
| 경품 응모 | 응모 가능 → 응모 완료 → 추첨 대기 → 당첨/미당첨 → 발송 완료 | 응모권 차감 시점, 중복 응모, 당첨자 고지, 제세공과금 |
| 기프티콘 교환 | 교환 요청 → 재고 확인 → 포인트 차감 → 핀 발급 → 발송 성공/실패 → 재발송/환불 | 외부 벤더 장애, 품절, 유효기간, 번호 노출 보안 |
| 취소/환불 | 요청 접수 → 사용 여부 확인 → 회수 가능/불가 → 원장 보정 | 이미 사용된 쿠폰 처리, 유상/무상 포인트 환급 순서 |
| 부정사용 | 이상 패턴 감지 → 보류 → 소명 → 지급/회수 | 다계정, 자동화, 추천인 어뷰징, 위치 조작 |

### 기프티콘 교환 상태 예시

```text
requested -> inventory_checked -> points_reserved -> pin_issued
-> sent -> delivered
```

실패 흐름도 반드시 정의해야 합니다.

```text
pin_issue_failed -> points_released
send_failed -> resend_pending
vendor_timeout -> manual_review
used_coupon_refund_denied
```

> **정책 메모**
> 모바일상품권의 유효기간, 기간 연장, 환급, 무상 제공 상품권 예외는 구분해서 설계해야 합니다. 경품 응모는 표시·광고, 개인정보 처리, 제세공과금 이슈가 얽힐 수 있어 캠페인 전 법무 검토 대상으로 분리하는 것이 좋습니다.

**챕터 요약:** 응모와 교환은 UX보다 상태, 고지, 재고, 외부 API 실패 처리가 먼저 설계돼야 합니다.

## 6. 온체인·오프체인 데이터 흐름

**핵심 메시지:** 모든 데이터를 온체인에 올리는 것이 좋은 Web3가 아닙니다. 개인정보와 운영 데이터는 오프체인, 검증이 필요한 권리와 거래 증거만 온체인 후보로 봅니다.

```text
사용자 행동
-> 앱 서버 이벤트 수집
-> 부정사용·조건 검증
-> 내부 재화 원장 반영
-> 필요한 경우 스마트 컨트랙트 호출
-> 블록 탐색기 거래 해시 생성
-> 앱 내 거래내역/CS 도구에 원장 ID와 해시 연결
```

| 데이터 | 권장 위치 | 이유 |
|---|---|---|
| 이름, 휴대폰, 배송지, 기프티콘 수신번호 | 오프체인 | 개인정보 삭제·정정·보관기간 관리 필요 |
| 포인트 지급/차감 상세 | 보통 오프체인 원장 | 빈번한 거래와 CS 처리에 유리 |
| NFT/SBT, 토큰 지급, 공개 인증 | 선택적 온체인 | 공개 검증·이전 가능성·외부 활용이 필요할 때 |
| 거래 해시, 컨트랙트 주소 | 온체인 + 오프체인 연결 | 탐색기 검증과 내부 CS 추적을 동시에 지원 |

### 온체인화 판단 기준

온체인 후보가 되려면 아래 질문에 답할 수 있어야 합니다.

1. 외부에서 공개 검증해야 하는가
2. 다른 서비스에서 재사용해야 하는가
3. 수수료와 처리 지연을 감수할 가치가 있는가
4. 개인정보나 민감 정보가 노출되지 않는가
5. 잘못 기록됐을 때 정정 또는 보정 정책이 있는가

> **개인정보·위치정보**
> 취향 데이터, 광고 반응, 위치 기반 걷기/방문 미션은 동의, 보관기간, 제3자 제공, 철회 기능을 별도 설계해야 합니다. 개인정보 처리방침과 위치정보 관련 법령은 출시 전 확인 대상입니다.

**챕터 요약:** 온체인은 검증용, 오프체인은 운영용이라는 기준을 잡으면 개인정보와 CS 리스크를 줄일 수 있습니다.

---

## 7. 운영·CS·리스크 관리

**핵심 메시지:** 월렛 서비스의 품질은 문제가 생겼을 때 어디까지 설명하고 복구할 수 있는가에서 드러납니다.

| 상황 | 사용자 안내 | 내부 확인 필드 |
|---|---|---|
| 전송 지연 | 네트워크 혼잡, 컨펌 대기, 탐색기 링크 안내 | chain, tx_hash, nonce, status, gas_fee |
| 오입금/오전송 | 지원 체인/주소 여부 확인, 복구 가능성 제한 고지 | deposit_address, chain, asset, custody_access |
| 포인트 미지급 | 미션 완료 조건과 광고주 승인 대기 안내 | campaign_id, user_id, event_id, fraud_score |
| 쿠폰 미수신 | 발송 번호, 벤더 상태, 재발송 가능 여부 확인 | vendor_order_id, pin_status, sent_at |
| 계정·기기 변경 | 키 관리 모델에 따른 복구 범위 고지 | auth_method, wallet_address, recovery_status |
| 부정사용 의심 | 임시 보류와 소명 절차 안내 | device_id, ip, referral_graph, velocity_rule |

### 운영자가 조회할 수 있어야 하는 기본 필드

| 영역 | 필드 |
|---|---|
| 사용자 | user_id, 가입일, 인증 방식, 계정 상태 |
| 월렛 | wallet_address, chain, 생성 방식, 연결 상태 |
| 온체인 거래 | tx_hash, chain, asset, amount, status, explorer_url |
| 내부 원장 | ledger_id, balance_before, balance_after, reason_code |
| 외부 주문 | vendor_order_id, vendor_status, request_at, response_at |
| 리스크 | fraud_score, rule_id, hold_reason, reviewer_id |

**챕터 요약:** 출시 전 운영표를 먼저 만들면 개발 범위도 또렷해집니다. “이 화면을 만든다”보다 “이 상태를 처리한다”가 더 정확한 요구사항입니다.

## 8. 출시 전 체크리스트

### 월렛 정책

- [ ] 키 관리 주체가 명확하다
- [ ] 복구 방식과 복구 불가 상황이 명시되어 있다
- [ ] 지원 체인과 미지원 체인이 구분되어 있다
- [ ] 주소 생성, 연결, 비활성화 정책이 있다
- [ ] 수수료 부담 주체와 무료 거래 한도가 정해져 있다

### 거래와 서명

- [ ] 전송, 수신, 서명, 승인, 승인 취소가 각각 정의되어 있다
- [ ] 사용자에게 보여줄 서명 문구가 준비되어 있다
- [ ] 고위험 권한 요청은 별도 경고를 제공한다
- [ ] 거래 해시와 내부 원장 ID를 함께 추적한다

### 재화와 원장

- [ ] 포인트, 응모권, 쿠폰, 토큰, NFT를 같은 재화로 뭉개지 않았다
- [ ] 지급 근거, 차감 순서, 만료, 회수 정책이 있다
- [ ] 원장 상태값과 운영자 보정 기준이 있다
- [ ] 유상/무상 재화의 환불 기준이 구분되어 있다

### 운영과 리스크

- [ ] 전송 지연, 오입금, 미지급, 쿠폰 실패 대응 문구가 있다
- [ ] 운영자가 조회해야 할 필드가 정리되어 있다
- [ ] 부정사용 탐지 후 보류, 소명, 해제, 회수 절차가 있다
- [ ] 법무, 개인정보, 위치정보, 소비자보호 검토 항목이 분리되어 있다

## 빠른 용어 사전

| 용어 | 한 줄 뜻 |
|---|---|
| 키 관리 주체 | 개인키 또는 복구 권한을 누가 책임지는지 |
| 커스터디 | 서비스 회사가 키 또는 자산 접근 권한을 관리하는 방식 |
| 셀프 커스터디 | 사용자가 직접 키와 복구 수단을 관리하는 방식 |
| 계정 추상화 | 지갑 계정에 복구, 한도, 자동화, 수수료 대납 같은 정책을 넣는 방향 |
| Paymaster | 사용자의 가스비를 대신 내는 주체 |
| 세션키 | 제한된 기간과 권한만 가진 임시 키 |
| 내부 원장 | 서비스가 포인트, 쿠폰, 응모권 등 재화 변동을 관리하는 장부 |
| 거래 해시 | 온체인 거래를 조회할 수 있는 식별값 |
| 상태 머신 | 요청, 대기, 성공, 실패, 취소처럼 상태 전이를 정의한 구조 |
| 오프체인 | 블록체인 밖의 서버나 DB에 저장하는 방식 |
| 온체인 | 블록체인에 기록하는 방식 |
